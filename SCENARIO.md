# Match-Day Scenario: What the Assistant Surfaces and Why

This walkthrough uses the **actual** recommendation rules in `frontend/src/lib/assistantEngine.ts` (mirrored in `backend/src/lib/assistantEngine.ts`). The congestion and incident cases below match concrete strings the engine emits for the stated inputs. The delay/bottleneck cases are constructed inputs so a reader can see the full rule family without depending on whatever the live simulator seeded at clone time.

---

## Starting picture (T−0)

Live ops console is open. One fixture is still live deep into the second half; a later slot on the same venue is already delayed; West Gate telemetry is rising; two Gate B alerts sit unacknowledged.

| Signal | Value used by the engine |
| --- | --- |
| Live match | `M-101` — REAL MADRID vs MANCHESTER CITY, `status: live`, `timeElapsed: 82` |
| Congested zone | `SECTOR GATE B (WEST)` — occupancy `4,550 / 5,000` (**91%**) |
| Unresolved alerts | (1) `"Gate B access congestion detected…"` warning; (2) `"Gate B secondary turnstile queue…"` warning — both `isAcknowledged: false` |
| Bracket delay (constructed for the delay-risk example below) | Delayed `QF-3` + scheduled `QF-4` at `MAIN VENUE Arena`, **45-minute** kickoff gap |
| Downstream bracket | Semifinal still shows a `TBD` side waiting on an unresolved quarterfinal |

Operators see live numbers on the Dashboard and Stadium views; the **Operations Assistant** drawer lists prioritized dispatches generated by `generateRecommendations()`.

---

## What fires, in priority order

### 1. Gate congestion — `evaluateGateCongestion`

**Why it triggers:** Any live match with `timeElapsed >= 75`, and a zone with occupancy ratio `>= 0.85`.

**Exact input → output for this scenario:**

- Input: zone at 91%, `M-101` at minute 82 → **8 minutes** remaining (`90 - 82`).
- Output recommendation:
  - **Priority:** `high` (91% is below the 95% critical band)
  - **Title:** `Gate Congestion Risk: SECTOR GATE B (WEST)`
  - **Reasoning includes:**  
    `Zone "SECTOR GATE B (WEST)" occupancy is at 91% (4,550/5,000)`  
    `Related live match M-101 (REAL MADRID vs MANCHESTER CITY) ends in 8 min`
  - **Suggested action:** `Open overflow routing or gates for SECTOR GATE B (WEST) immediately.`

If the same zone were still at 91% but the match were at minute 70, **this rule would emit nothing** — timing context matters, not occupancy alone.

### 2. Incident cluster — `evaluateIncidentEscalation`

**Why it triggers:** At least **two** unacknowledged alerts that keyword-map to the same zone (`Gate B` / `WEST` → `SECTOR GATE B (WEST)`).

**Exact input → output:**

- Input: the two Gate B warnings above.
- Output recommendation:
  - **Priority:** `high` (no `critical`-level alert in the cluster)
  - **Title:** `Incident Dispatch: SECTOR GATE B (WEST)`
  - **Reasoning includes:** count of unacknowledged alerts in that zone, recent-count / highest severity line, then each alert message
  - **Suggested action:** `Dispatch a tactical operations patrol to resolve incidents in SECTOR GATE B (WEST).`

Acknowledging either alert drops the cluster below two and **clears** this recommendation on the next evaluation — again contextual, not sticky.

### 3. Schedule delay risk — `evaluateMatchDelayRisk`

**Why it triggers:** A `delayed` bracket match shares a venue with at least one `scheduled` match.

**Constructed input → output** (same functions the app runs; numbers chosen so a reader can replay them):

- Input: delayed `QF-3` and scheduled `QF-4` at `MAIN VENUE Arena`, with scheduled starts **45 minutes** apart.
- Output recommendation:
  - **Priority:** `high` (gap `<= 60` minutes → tight turnaround)
  - **Title:** `Schedule Delay Risk: MAIN VENUE Arena`
  - **Reasoning includes:** which match is delayed, which subsequent match shares the venue, and `Next same-venue kickoff is 45 min after the delayed match slot`
  - **Suggested action:** `Adjust subsequent kickoff slot for QF-4 by +30 minutes.`

A three-hour gap would still surface a recommendation, but at **`low`** priority — same rule family, different urgency from context.

### 4. Bracket bottleneck — `evaluateTournamentBottleneck` (if TBD slots remain)

**Why it triggers:** A delayed match in an earlier round while a later round still has `TBD` placeholders.

**Exact shape of the output:**

- **Title:** `Bracket Bottleneck: <round name>` (e.g. quarterfinals)
- **Priority:** `high` only when the immediate next round still has TBD **and** the current round is already ≥ 75% complete; otherwise `medium`
- **Suggested action:** fast-track pitch resolution or manually override the winner so the bracket can advance

---

## What the operator does next

1. Opens the Assistant, expands reasoning (deterministic trail first; optional Gemini **AI Summary** if a key is configured, otherwise a local template sentence).
2. Accepts or dismisses each dispatch — JWT role must be `admin` or `operator`; `viewer` can read but cannot mutate.
3. Each Accept/Dismiss is written to the **Decision Log** audit trail.
4. Live state changes (occupancy drop, alert ACK, delay cleared) cause the next `generateRecommendations()` pass to drop or re-prioritize items automatically.

---

## Why this demonstrates “logical decision making”

The assistant does not invent a narrative. Each card is the output of a named function over a concrete state vector. Changing one contextual input (match minute, ACK flag, kickoff gap, TBD presence) changes the catalog in a way an evaluator can reproduce from the source without trusting a slide deck.
